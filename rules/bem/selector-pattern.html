@morev/bem/selector-pattern ‚Äã

Enforces naming patterns for BEM entities.

You can define specific patterns for:

EntityExample
BEM block.the-component__element--theme--dark
BEM element.the-component__element--theme--dark
BEM modifier name.the-component__element--theme--dark
BEM modifier value.the-component__element--theme--dark

The rule also supports:

‚úÖ Predefined keywords for common naming styles ('KEBAB_CASE', 'PASCAL_CASE', etc.)
‚úÖ Wildcards inside string patterns - 'foo-*'
‚úÖ Multiple patterns per entity
‚úÖ Complete disallowing of modifier values
‚úÖ Ignoring specific BEM blocks
‚úÖ Custom error messages

Rule options ‚Äã

All options are optional and have sensible default values.

js
// üìÑ .stylelintrc.js

export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': true,
  }
}
js
// üìÑ .stylelintrc.js

export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      patterns: {
        block: 'KEBAB_CASE',
        element: /^[a-z][0-9a-z]*(?:-[0-9a-z]+)*$/,
        modifierName: 'KEBAB_CASE',
        modifierValue: 'KEBAB_CASE',
      }
      ignoreBlocks: ['swiper-*', 'u-*'],
      separators: {
        element: '__',
        modifier: '--',
        modifierValue: '--',
      },
      messages: {
        block: (name, fullSelector, patterns) => {
          return `Unexpected block name ${name}`;
        },
      },
    }],
  },
}

  /**
   * Object containing allowed patterns for different BEM entities.

     *

     */
    modifierName?: string | RegExp | Array<string | RegExp>;

     * or keywords like `KEBAB_CASE`. \
     * Use `false` to forbid modifier values entirely.
     *
     * @default KEBAB_CASE_REGEXP

   * This allows the rule to work correctly with non-standard BEM naming conventions.
   */
   */

     *

     */
    element?: string;

    /**
     * String used as the BEM modifier separator.

    modifier?: string;

    block?: (name: string, fullSelector: string, patterns: ProcessedPattern[]) => string;

     *

    element?: (name: string, fullSelector: string, patterns: ProcessedPattern[]) => string;

     * @param   fullSelector   Full resolved BEM selector for the violation.

     *
     * @returns                Error message.

    modifierName?: (name: string, fullSelector: string, patterns: ProcessedPattern[]) => string;

     * @returns                Error message.

    modifierValue?: (name: string, fullSelector: string, patterns: ProcessedPattern[] | false) => string;

patterns ‚Äã

All of the keys of patterns property define the allowed naming patterns for different parts of your selectors.

Note

The options shown below are not defaults or best practices.
They simply demonstrate the flexibility of configuration.

Each option may accept:

string

Interpreted as a RegExp in string form (if enclosed in / with optional flags), as a pattern supporting wildcards, or a predefined keyword representing a popular pattern preset.

js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      patterns: {
        block: 'component-*', 
        element: 'KEBAB_CASE', 
        modifierName: 'is-*', 
      },
    },
  },
}
scss
// ‚úÖ Block name matches the pattern `'component-*'`
// ‚úÖ Element name matches the pattern `KEBAB_CASE`
// ‚úÖ Modifier name matches the pattern `is-*`
.component-foo__some-element--is-active {}

// ‚ùå Block name does not match the pattern ``'component-*'`
// ‚úÖ Element name matches the pattern `KEBAB_CASE`
// ‚úÖ Modifier name matches the pattern `is-*`
.the-component__element--is-active {}

// ‚úÖ Block name matches the pattern `'component-*'`
// ‚úÖ The element is absent and not validated
// ‚ùå Modifier name does not match the pattern `is-*`
.component-foo--active {}

// ‚úÖ Block name matches the pattern `'component-*'`
// ‚ùå The element does not match the pattern `KEBAB_CASE`
// ‚úÖ Modifier name is absent and not validated
.component-foo__FooElement {}

The plugin provides common naming presets as convenient keywords.

KeywordDescriptionExported RegExp Name

The corresponding regular expressions are also exported from the package for use in custom tooling using a named export from /constants:

ts
import {
  KEBAB_CASE_REGEXP, PASCAL_CASE_REGEXP, CAMEL_CASE_REGEXP, SNAKE_CASE_REGEXP
} from '@morev/stylelint-plugin/constants';
RegExp

A regular expression for advanced matching, if the configuration format allows it.
For example, yaml and json configurations do not support direct RegExp usage - use string-based format described above if your config is written using such formats.

js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      patterns: {
        blockPattern: /^component-[a-z-]+/, 
      },
    },
  },
}
scss
// ‚úÖ Block name matches the pattern `/^component-[a-z-]+/`
.component-foo__element {}

// ‚ùå Block name does not match the pattern `/^component-[a-z-]+/`
.foo-component__element {}

TIP

The package provides several popular patterns as named exports from /constants, in case you want to use them in your own tooling.

ts
import {
  KEBAB_CASE_REGEXP, PASCAL_CASE_REGEXP, CAMEL_CASE_REGEXP, SNAKE_CASE_REGEXP
} from '@morev/stylelint-plugin/constants';
array

A list of any combination of the above.

js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      patterns: {
        modifierName: ['is-*', /.*foo.*/], 
      }
    },
  },
}
scss
// ‚úÖ Modifier name matches the pattern `'is-*'`
.the-component--is-active {}

// ‚úÖ Modifier name match the pattern `/.*foo.*/`
.the-component--foo-bar {}
.the-component--bar-foo {}

// ‚úÖ Modifier name is absent and not validated
.the-component__element {}

// ‚ùå Modifier name does not match any of the defined patterns
.the-component--baz {}

Disabling modifier values ‚Äã

Some teams prefer a simplified BEM structure where modifiers are always flat - that is, modifier names encode their full meaning without a separate value part.

scss
// Instead of
.block--theme--dark {}

// ...they write:
.block--theme-dark {}

This approach enforces a strict three-level model:

  1. block
  2. element
  3. modifier

It avoids introducing a fourth, optional level (modifier value) which can complicate mental parsing and tooling logic.

To enforce this model, set the modifierValuePattern to false.

js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      patterns: {
        modifierValue: false, 
      }
    }],
  },
}

This will disallow any BEM selector that contains a separate modifier value.
If you're using flattened values, they will be treated as part of the modifier name, and validated against pattern.modifierName.

Example ‚Äã
scss
// ‚úÖ Valid: modifier is flattened into one name
.block--theme-dark {}

// ‚ùå Invalid: separate modifier name + value
.block--theme--dark {}

separators ‚Äã

The rule supports different naming conventions for BEM entities by allowing you to configure the separators between block elements, modifiers, and modifier values.

This flexibility ensures compatibility with all popular BEM styles described in the official BEM methodology naming convention or even custom ones.

Available separators ‚Äã

OptionDefaultDescription
element__Separator between block and element.
modifier--Separator between block/element and modifier name.
modifierValue--Separator between modifier name and modifier value.

Configuration examples ‚Äã

Two Dashes style (default)
js
export default {

  rules: {

      separators: {
        element: '__',
        modifier: '--',
        modifierValue: '--',
      },
    },
  },
}

.block__element--modifier-name {}
.block__element--modifier-name--value {}

.block--modifier {}
.block--modifier-name--value {}

Rule is separator-agnostic

You can adapt this rule to any BEM naming convention using the available options.
For details on naming principles, refer to the official BEM methodology guide.

ignoreBlocks ‚Äã

In some projects, not all class names are strictly controlled by your codebase or team conventions. For example, you might need to override:

  • Classes generated by third-party libraries.
  • Global or legacy utility blocks.
  • System-wide technical classes that don't follow your BEM patterns.

Configuration examples ‚Äã

Ignore by exact name
js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      ignoreBlocks: ['legacy-component'] 
    },
  },
}
scss
// ‚úÖ All selectors related to the .legacy-component block are completely ignored
.legacy-component.FOO_b-a-r {}
.legacy-component__ELEMENT {}
.legacy-component--MoDiFiEr {}
Ignore by wildcard pattern
js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      ignoreBlocks: ['swiper-*'] 
    },
  },
}
scss
// ‚úÖ All selectors that begin with `.swiper-` are completely ignored
.swiper-slide.swiper-slide-active {}
Ignore by regular expression
js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      ignoreBlocks: [/.*foo.*/] 
    },
  },
}
scss
// ‚úÖ All blocks containing `foo` are completely ignored
.Block-foo__Do-WHATEVER_you-want-here {}
.foo.IS_ACTIVE {}

Notes ‚Äã

  • The rule matches ignoreBlocks only against the block part of the selector.
  • If the block name matches, the entire selector is excluded from validation.
  • You can mix plain strings, wildcards, and regular expressions in the same list.

messages ‚Äã

The rule provides built-in error messages for all violations it detects.
You can customize them using the messages option. This can be useful to:

  • Adjust the tone of voice to match your team's style;
  • Translate messages into another language;
  • Provide additional project-specific context or documentation links.

INFO

You don't need to override all message functions ‚Äî or any of them at all.

Each message function receives the detected entity name, full resolved BEM selector and the list of expected patterns.

Example ‚Äã

js
export default {
  plugins: ['@morev/stylelint-plugin'],
  rules: {
    '@morev/bem/selector-pattern': [true, {
      messages: {
        block: (name, fullSelector, patterns) =>
          `‚õî Block name "${name}" is invalid. It must follow: ${patterns.map(p => p.source).join(', ')}`,

        element: (name) =>
          `‚ö†Ô∏è Element "${name}" doesn't look good.`,

        modifierValue: (name, fullSelector, patterns) => {
          if (patterns === false) {
            return `Modifier values in "${fullSelector}" are completely forbidden.`;
          }

          return `Modifier value "${name}" must match: ${patterns.map(p => p.source).join(', ')}`;
        },
      },
    }],
  },
}

How message formatting works

If your custom message function returns anything other than a string (e.g., undefined), the rule will automatically fall back to the default built-in message.

Additionally, all custom messages are automatically processed through stripIndent function, so it's safe and recommended to use template literals (backticks, `) for multiline messages without worrying about inconsistent indentation.
